Problem 10: Detect Cycle in Undirected Graph

[Algorithm Family: DFS — Cycle Detection]

[Phase 2: Core Patterns — "I know WHICH traversal and WHY"]

PROBLEM STATEMENT

You are a network engineer. Given N computers (numbered 0 to N-1)
and cables connecting them, you need to determine:

  Does the network contain a CYCLE?

A cycle means: starting from some computer, you can follow cables
and return to the SAME computer without using any cable twice.

Why this matters:
  → Cycles in networks can cause infinite routing loops
  → Cycles mean "redundant connections" — more cables than needed
  → A tree (no cycles) with V nodes has exactly V-1 edges
  → If edges ≥ V → guaranteed cycle (for connected graph)

Input:
  n = 5  (computers: 0, 1, 2, 3, 4)
  edges = [[0,1], [1,2], [2,3], [3,4], [4,1]]

Output:
  True — cycle exists: 1 → 2 → 3 → 4 → 1

Seeing the Cycle

  Starting at node 1:

STEP 2: BRUTE FORCE FIRST

  Naive idea: "For every edge (u, v), remove it from the graph.
               Then check if u can still reach v (using BFS/DFS).
               If yes → that edge is part of a cycle."
  
  → E edges to try
  → Each reachability check: O(V + E)
  → Total: O(E × (V + E))
  
  For V = 100,000 and E = 200,000:
    → 200,000 × 300,000 = 6 × 10¹⁰ → TOO SLOW

  Another naive: "Count edges. If E >= V → cycle exists"
  → This only works for CONNECTED graphs!
  → Disconnected graph can have E < V with no cycles
     (multiple small trees)
  → Not a general solution.

Why Brute Force Fails

  We need a single-pass O(V + E) algorithm.
  
  KEY INSIGHT from DFS (Problem 4):
    → DFS explores one path deeply
    → If during DFS, we encounter a node that's ALREADY VISITED...
    → ...that means we found a "back edge" — a shortcut back to
      a node we've already been to
    → This shortcut creates a CYCLE!
    
  But there's a TRAP in undirected graphs...

STEP 3: IDENTIFYING THE GRAPH PATTERN

The Trap: Every Undirected Edge Looks Like a "Cycle"

  Consider the simple graph:
    0 --- 1
    
  Adjacency list:
    0: [1]
    1: [0]

  DFS from 0:
    Visit 0 (mark visited)
    → Neighbor 1: not visited → DFS(1)
      Visit 1 (mark visited)
      → Neighbor 0: ALREADY VISITED!
      
  Does this mean cycle? NO!
  
  0 and 1 are connected by ONE edge.
  When we go from 0 to 1, and then look at 1's neighbors,
  we see 0 again — but that's just the SAME EDGE we came from!
  
  In undirected graphs:
    Edge 0—1 appears as:
      0's neighbor list contains 1
      1's neighbor list contains 0
    
  So when DFS goes 0→1, it sees 0 as 1's neighbor.
  But 0 is 1's PARENT in the DFS tree — not a cycle!
  
  ┌──────────────────────────────────────────────────────────────┐
  │  THE RULE:                                                    │
  │                                                               │
  │  When DFS at node u finds a visited neighbor v:               │
  │    → If v is u's PARENT → NOT a cycle (just the edge we came)│
  │    → If v is NOT u's parent → CYCLE FOUND!                   │
  │                                                               │
  │  "Visited AND not parent" = CYCLE                             │
  │  "Visited AND is parent"  = just the edge we traversed       │
  └──────────────────────────────────────────────────────────────┘

Visual: Parent vs Back Edge

  Graph with cycle:
    0 --- 1 --- 2
          |     |
          4 --- 3

  DFS from 0:
    Visit 0 → Visit 1 → Visit 2 → Visit 3 → Visit 4

  DFS tree:
    0 → 1 → 2 → 3 → 4

  At node 4, neighbors are [3, 1]:
    → 3: visited, IS parent (we came from 3) → NOT a cycle
    → 1: visited, NOT parent (we came from 3, not 1) → ★ CYCLE! ★

  The edge 4—1 is a BACK EDGE — it connects node 4 to an
  ANCESTOR (node 1) in the DFS tree.
  
  Back edge = cycle!

  ┌──────────────────────────────────────────────────────────┐
  │  DFS Tree:    0 → 1 → 2 → 3 → 4                        │
  │                    ↑_____________│   ← back edge 4→1     │
  │                                                          │
  │  Cycle: 1 → 2 → 3 → 4 → 1                              │
  └──────────────────────────────────────────────────────────┘

Visual: No Cycle (Tree Graph)

  Tree graph:
    0 --- 1 --- 2
          |
          3 --- 4

  DFS from 0:
    Visit 0 → Visit 1 → Visit 2 (dead end, backtrack)
                       → Visit 3 → Visit 4 (dead end, backtrack)

  DFS tree:
    0 → 1 → 2
         → 3 → 4

  At each node, the ONLY visited neighbor is the PARENT:
    Node 2: neighbor 1 visited, IS parent → no cycle
    Node 4: neighbor 3 visited, IS parent → no cycle
    Node 3: neighbor 1 visited, IS parent → no cycle

  No back edges → NO CYCLE ✓

Why DFS (Not BFS) for Cycle Detection?

  BFS CAN detect cycles in undirected graphs:
    → If you discover a neighbor that's already visited 
      AND not your parent → cycle
    → Works the same way conceptually
    
  But DFS is MORE NATURAL for cycle detection because:
    1. DFS naturally creates a TREE structure (DFS tree)
    2. Back edges (non-tree edges to ancestors) = cycles
    3. This extends cleanly to DIRECTED graphs (Problem 11)
    4. DFS is the STANDARD approach taught and expected
    
  For undirected graphs: both BFS and DFS work
  For directed graphs: ONLY DFS works cleanly (Problem 11)
  
  We'll show DFS as the primary approach.

STEP 4: ALGORITHM WALKTHROUGH — Step by Step

The Algorithm (in English)

  DFS(node, parent):
    1. Mark node as visited
    2. For each neighbor of node:
       a. If neighbor is NOT visited → recurse DFS(neighbor, node)
          → If recursion returns True (cycle found deeper) → return True
       b. If neighbor IS visited AND neighbor ≠ parent → CYCLE FOUND!
          → Return True
    3. If all neighbors checked and no cycle → return False

  Main:
    For each unvisited node (handle disconnected graphs — Problem 5):
      If DFS(node, parent=-1) returns True → cycle exists
    If no DFS found a cycle → no cycle

STEP 5: COMPLETE CODE

Version 1: DFS Recursive

from collections import defaultdict

def has_cycle_undirected_dfs(n, edges):
    """
    Detect cycle in an UNDIRECTED graph using DFS.
    
    Key insight:
        During DFS, if we find a visited neighbor that is NOT our parent,
        we've found a back edge → cycle exists.
    
    Why parent check is needed:
        In undirected graph, edge u—v appears in BOTH adjacency lists.
        When DFS goes u→v, then v sees u as neighbor.
        u IS visited, but u is v's PARENT — not a cycle!
        Without parent check: every edge looks like a cycle → WRONG.
    
    Why this is DIFFERENT from directed graphs (Problem 11):
        Directed: edge u→v only appears in u's list, not v's.
        No "parent" ambiguity in directed graphs.
        But directed cycles need 3-state coloring instead.
    
    Args:
        n: number of nodes
        edges: list of [u, v] pairs (undirected)
    
    Returns:
        True if cycle exists, False otherwise
    """
    # ─── BUILD GRAPH (Problem 1) ───
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)
    
    # ─── DFS WITH PARENT TRACKING ───
    visited = set()
    
    def dfs(node, parent):
        """
        Returns True if a cycle is found in the component containing node.
        
        Args:
            node: current node being explored
            parent: the node we came from (-1 if root)
        """
        visited.add(node)
        
        for neighbor in graph[node]:
            if neighbor not in visited:
                # Unvisited neighbor → explore deeper
                if dfs(neighbor, node):     # pass current node as parent
                    return True             # cycle found deeper → propagate
            
            elif neighbor != parent:
                # Visited neighbor that's NOT our parent → CYCLE!
                # This is a BACK EDGE
                return True
        
        # All neighbors checked, no cycle found from this node
        return False
    
    # ─── CHECK ALL COMPONENTS (Problem 5 pattern) ───
    # Graph might be disconnected — need to check each component
    for node in range(n):
        if node not in visited:
            if dfs(node, -1):      # -1 means no parent (root of DFS)
                return True
    
    return False


# ─── TESTS ───
# Graph WITH cycle
print(has_cycle_undirected_dfs(5, [[0,1],[1,2],[2,3],[3,4],[4,1]]))
# Output: True (cycle: 1-2-3-4-1)

# Tree (no cycle)
print(has_cycle_undirected_dfs(5, [[0,1],[1,2],[1,3],[3,4]]))
# Output: False

# Disconnected, one component has cycle
print(has_cycle_undirected_dfs(6, [[0,1],[1,2],[2,0],[3,4],[4,5]]))
# Output: True (cycle: 0-1-2-0)

# Disconnected, no cycles
print(has_cycle_undirected_dfs(6, [[0,1],[2,3],[4,5]]))
# Output: False (three separate edges, no cycles)

# Single node
print(has_cycle_undirected_dfs(1, []))
# Output: False

# Two nodes, one edge
print(has_cycle_undirected_dfs(2, [[0,1]]))
# Output: False

Version 2: BFS Approach (Alternative)

from collections import defaultdict, deque

def has_cycle_undirected_bfs(n, edges):
    """
    Detect cycle in undirected graph using BFS.
    
    Same logic as DFS version:
        → Track parent for each node
        → If we discover a visited neighbor that's NOT parent → cycle
    
    BFS tracks parent via a parent dictionary (since BFS doesn't
    have a natural parent from recursion).
    
    Less common than DFS for cycle detection, but works correctly.
    """
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)
    
    visited = set()
    
    def bfs(source):
        """BFS from source, returns True if cycle found."""
        queue = deque([source])
        visited.add(source)
        parent = {source: -1}
        
        while queue:
            current = queue.popleft()
            
            for neighbor in graph[current]:
                if neighbor not in visited:
                    visited.add(neighbor)
                    parent[neighbor] = current
                    queue.append(neighbor)
                
                elif neighbor != parent[current]:
                    # Visited and NOT parent → CYCLE
                    return True
        
        return False
    
    # Check all components
    for node in range(n):
        if node not in visited:
            if bfs(node):
                return True
    
    return False


# ─── TESTS ───
print(has_cycle_undirected_bfs(5, [[0,1],[1,2],[2,3],[3,4],[4,1]]))  # True
print(has_cycle_undirected_bfs(5, [[0,1],[1,2],[1,3],[3,4]]))        # False

Version 3: Iterative DFS (No Recursion Limit)

from collections import defaultdict

def has_cycle_undirected_iterative(n, edges):
    """
    Iterative DFS cycle detection.
    
    Use when n > 1000 (recursion limit concern from Problem 4).
    
    Stack stores (node, parent) pairs so we know who discovered each node.
    """
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)
    
    visited = set()
    
    for start in range(n):
        if start in visited:
            continue
        
        # Stack: (node, parent)
        stack = [(start, -1)]
        visited.add(start)
        
        while stack:
            node, parent = stack.pop()
            
            for neighbor in graph[node]:
                if neighbor not in visited:
                    visited.add(neighbor)
                    stack.append((neighbor, node))
                
                elif neighbor != parent:
                    return True      # cycle found!
    
    return False


# ─── TESTS ───
print(has_cycle_undirected_iterative(5, [[0,1],[1,2],[2,3],[3,4],[4,1]]))  # True
print(has_cycle_undirected_iterative(5, [[0,1],[1,2],[1,3],[3,4]]))        # False

⚠️ Important Subtlety with Iterative DFS

  The iterative version has a subtle issue with the parent check.
  
  Problem: When a node is pushed multiple times before being popped,
  the "parent" stored with each push might differ.
  
  Example:
    0 --- 1 --- 2
    |           |
    +-----------+
    
  Node 2 might be pushed by BOTH node 1 and node 0.
  When we pop the second push, "parent" might be wrong.
  
  The visited-on-push approach (as shown) handles this because:
    → Node 2 is marked visited when FIRST pushed
    → Second push attempt: 2 is already visited
    → If 2 is visited and not parent of current node → cycle detected
    
  For most cases this works correctly. For complete correctness
  with iterative DFS, using a parent dictionary is safer:
def has_cycle_iterative_safe(n, edges):
    """Safer iterative version using parent dictionary."""
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)
    
    visited = set()
    
    for start in range(n):
        if start in visited:
            continue
        
        parent = {start: -1}
        stack = [start]
        visited.add(start)
        
        while stack:
            node = stack.pop()
            
            for neighbor in graph[node]:
                if neighbor not in visited:
                    visited.add(neighbor)
                    parent[neighbor] = node
                    stack.append(neighbor)
                
                elif neighbor != parent[node]:
                    return True
    
    return False

Version 4: Edge Count Quick Check (Connected Graph Only)

def has_cycle_edge_count(n, edges, connected=True):
    """
    Quick mathematical check (only for CONNECTED graphs):
    
    A TREE with V nodes has exactly V-1 edges.
    A connected graph with V nodes and E ≥ V edges MUST have a cycle.
    
    This is O(1) — just compare counts!
    
    LIMITATION: Only works if graph is known to be connected.
    For disconnected graphs: must use DFS/BFS approach.
    """
    if not connected:
        raise ValueError("This shortcut only works for connected graphs")
    
    return len(edges) >= n    # E >= V means cycle exists


# For a CONNECTED graph:
# V nodes, V-1 edges → tree (no cycle)
# V nodes, V edges → exactly one cycle
# V nodes, V+k edges → k+1 cycles (roughly)

print(has_cycle_edge_count(5, [[0,1],[1,2],[2,3],[3,4]]))        # False (4 edges, 5 nodes)
print(has_cycle_edge_count(5, [[0,1],[1,2],[2,3],[3,4],[4,1]]))  # True (5 edges, 5 nodes)

STEP 6: COMPLEXITY ANALYSIS

Time Complexity: O(V + E)

  WHY O(V + E)?
  
  Same as basic DFS (Problem 4):
    → Each node visited ONCE: O(V)
    → Each edge checked ONCE from each endpoint: O(E)
    → Parent comparison: O(1) per neighbor check
    → Total: O(V + E)

  With outer loop for disconnected graphs (Problem 5):
    → Same O(V + E) total — visited set prevents re-processing
    → Outer loop just distributes work across components

  COMPARE WITH BRUTE FORCE:
    Brute force: O(E × (V + E))
    DFS:         O(V + E)
    Speedup:     ~E times faster (for large E, this is massive)

Space Complexity: O(V)

  1. Adjacency list: O(V + E)    (input)
  2. Visited set: O(V)
  3. Recursion stack: O(V)        worst case (long chain)
  4. Parent parameter: O(1) per call (just one integer)
  
  Extra space: O(V)
  
  Iterative version: O(V) for stack + O(V) for parent dict = O(V)

STEP 7: THE PARENT CHECK — Deep Dive

Why Does Parent Check Work?

  In undirected graph, edge u—v creates TWO entries:
    graph[u] contains v
    graph[v] contains u
  
  When DFS goes from u to v:
    → v's parent = u
    → When DFS(v) checks neighbors, it sees u
    → u IS visited (we came from there)
    → But u is v's parent → this is the SAME edge we used to get here
    → NOT a cycle — just the reverse direction of the same edge
    
  If DFS(v) sees a visited neighbor w (where w ≠ parent u):
    → w was visited BEFORE we arrived at v
    → But w is NOT the node we came from
    → So there must be another path from w to v (through the DFS tree)
    → Plus the edge v—w (the one we just found)
    → Two different paths between w and v = CYCLE!
    
  ┌──────────────────────────────────────────────────────────────┐
  │  DFS Tree:    ... → w → ... → u → v                         │
  │                     ↑_______________│   ← back edge v—w      │
  │                                                              │
  │  Tree path:   w → ... → u → v     (through DFS tree)        │
  │  Back edge:   v → w               (the edge we just found)  │
  │  Cycle:       w → ... → u → v → w                           │
  └──────────────────────────────────────────────────────────────┘

What About Multi-Edges (Parallel Edges)?

  What if there are TWO edges between nodes 0 and 1?
    edges = [[0,1], [0,1]]
    
  Adjacency list:
    0: [1, 1]
    1: [0, 0]
    
  DFS(0, parent=-1):
    Visit 0
    Neighbor 1 (first): not visited → DFS(1, parent=0)
      Visit 1
      Neighbor 0 (first): visited, IS parent → skip
      Neighbor 0 (second): visited, IS parent → skip
    Return False
    Neighbor 1 (second): visited, IS parent... wait, 1 IS parent? No!
    0's parent is -1, not 1.
    1 is visited and 1 ≠ parent(-1) → CYCLE!
    
  Hmm, but this might detect it at a different point depending on order.
  
  Actually, multi-edges DO form cycles: 0→1→0 using two different edges.
  The simple parent check handles this MOST of the time.
  
  BUT there's an edge case: if node 0 and node 1 have exactly 2 parallel
  edges, the parent check might skip BOTH (treating both as parent).
  
  SAFE FIX for multi-edges: track parent EDGE INDEX, not parent NODE.
  
  For most interview problems: parallel edges don't appear.
  Simple parent check is sufficient.

STEP 8: WHAT IF THE GRAPH WERE...

What if Directed?

  ★ THE PARENT CHECK DOES NOT WORK FOR DIRECTED GRAPHS! ★
  
  Example:
    0 → 1 → 2
    
  DFS from 0: visit 0 → visit 1 → visit 2
  At node 2: no unvisited neighbors, no visited non-parent neighbors
  → No cycle detected ✓
  
  But consider:
    0 → 1
    0 → 2
    1 → 2
    
  DFS from 0: visit 0 → visit 1 → visit 2 (from 1)
  Back at 0: neighbor 2 is visited, 2 ≠ parent(-1)
  → Parent check says CYCLE!
  → But there's NO cycle! (0→1→2 and 0→2 are just two paths, not a cycle)
  
  In directed graphs:
    → A visited node might be reachable via multiple paths
    → That's NOT a cycle — it's just a DAG (directed acyclic graph)
    → A cycle requires a path BACK to a node currently being processed
    → Need 3-state coloring: WHITE, GRAY, BLACK
    → This is Problem 11
    
  ┌──────────────────────────────────────────────────────────────┐
  │  UNDIRECTED cycle detection: visited + parent check          │
  │  DIRECTED cycle detection:   3-state coloring (Problem 11)   │
  │                                                              │
  │  They are FUNDAMENTALLY DIFFERENT algorithms!                │
  │  Don't mix them up.                                          │
  └──────────────────────────────────────────────────────────────┘

What if Weighted?

  Weights don't affect cycle detection.
  A cycle is about STRUCTURE (can you return to a node?), not COST.
  
  Just ignore weights:
    for u, v, weight in edges:
        graph[u].append(v)     # ignore weight
        graph[v].append(u)

What if We Need to FIND the Cycle (Not Just Detect)?

  To find the actual cycle:
    → When back edge (node, ancestor) is found:
    → Trace from node back through parents until we reach ancestor
    → That trace + the back edge = the cycle
    
  cycle_nodes = []
  current = node
  while current != ancestor:
      cycle_nodes.append(current)
      current = parent[current]
  cycle_nodes.append(ancestor)
  cycle_nodes.reverse()
  # cycle_nodes now contains the cycle!

What if Disconnected?

  Already handled! (Problem 5 pattern)
  
  The outer loop:
    for node in range(n):
        if node not in visited:
            if dfs(node, -1):
                return True
  
  Checks EVERY component. A cycle in ANY component → return True.
  
  Example:
    Component 1: {0, 1, 2} — no cycle (tree)
    Component 2: {3, 4, 5, 6} — has cycle 3-4-5-3
    
  DFS on component 1: no cycle found
  DFS on component 2: cycle found → return True

STEP 10: CYCLE DETECTION — WHEN TO USE WHICH METHOD

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  METHOD 1: DFS + Parent Check (this problem)                      │
  │    → Undirected graphs                                            │
  │    → O(V + E) time, O(V) space                                    │
  │    → Standard interview approach                                   │
  │                                                                    │
  │  METHOD 2: 3-State DFS (Problem 11)                               │
  │    → Directed graphs                                              │
  │    → WHITE/GRAY/BLACK coloring                                    │
  │    → O(V + E) time, O(V) space                                    │
  │                                                                    │
  │  METHOD 3: Union-Find (Problem 29)                                │
  │    → Undirected graphs                                            │
  │    → Process edges one by one                                     │
  │    → If both endpoints in same set → cycle!                       │
  │    → O(E × α(V)) ≈ O(E) time                                     │
  │    → Elegant alternative to DFS for undirected                    │
  │                                                                    │
  │  METHOD 4: Edge Count (Version 4 above)                           │
  │    → Only for CONNECTED graphs                                    │
  │    → E ≥ V → cycle exists (for connected graph)                   │
  │    → O(1) time — instant!                                         │
  │    → But limited applicability                                    │
  │                                                                    │
  │  METHOD 5: Topological Sort (Problem 19/21)                       │
  │    → Directed graphs                                              │
  │    → If topo sort can't include all nodes → cycle exists          │
  │    → O(V + E) time                                                │
  │                                                                    │
  │  DECISION:                                                         │
  │    Undirected + simple check → DFS + parent (this problem)        │
  │    Undirected + edge-by-edge → Union-Find (Problem 29)            │
  │    Directed → 3-state DFS (Problem 11) or Topo Sort (Problem 21)  │
  │                                                                    │
  └──────────────────────────────────────────────────────────────────┘
