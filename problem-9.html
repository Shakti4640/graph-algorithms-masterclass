Problem 9: Shortest Path in Unweighted Graph

[Algorithm Family: BFS — Shortest Path]

[Phase 2: Core Patterns — "I know WHICH traversal and WHY"]

PROBLEM STATEMENT

You are navigating a maze-like building. Given N rooms 
(numbered 0 to N-1) and corridors connecting them, find the 
MINIMUM number of corridors you must walk through to get from
room SOURCE to room DESTINATION.

Every corridor takes exactly 1 minute to traverse.

If destination is unreachable, return -1.

Input:
  n = 8  (rooms: 0, 1, 2, 3, 4, 5, 6, 7)
  edges = [[0,1], [0,2], [1,3], [1,4], [2,5], [3,6], [4,6], [5,7], [6,7]]
  source = 0
  destination = 7

Output:
  3  (shortest path: 0 → 2 → 5 → 7, using 3 corridors)

  Other paths exist:
    0 → 1 → 3 → 6 → 7  (4 corridors — longer!)
    0 → 1 → 4 → 6 → 7  (4 corridors — longer!)
    
  Minimum is 3.

STEP 2: BRUTE FORCE FIRST

  Naive idea: "Find ALL possible paths from source to destination,
               then pick the shortest one"
  
  How to find all paths:
    → DFS with backtracking — explore every possible route
    → Track current path, when destination reached, record length
    → Compare all lengths, return minimum
  
  Problem:
    → Number of paths can be EXPONENTIAL
    → Complete graph with V nodes: up to (V-1)! paths
    → Even for moderate graphs: millions of paths
    → Time: O(V!) in worst case → IMPOSSIBLY SLOW
    
  Example: grid graph 10×10 = 100 nodes
    → Paths from corner to corner: astronomical number
    → Brute force would take longer than the age of the universe

Why DFS Fails for Shortest Path

  DFS finds A path, not necessarily the SHORTEST path.
  
  In our graph, DFS from 0 might explore:
    0 → 1 → 3 → 6 → 7    (found path of length 4)
    
  DFS doesn't know that 0 → 2 → 5 → 7 (length 3) exists
  unless it tries ALL paths and compares.

  ┌──────────────────────────────────────────────────────────────┐
  │  DFS explores ONE path deeply before trying alternatives     │
  │  → May find a LONG path first                                │
  │  → Must explore ALL paths to guarantee shortest              │
  │  → Exponential time!                                         │
  │                                                              │
  │  BFS explores ALL paths of length 1, then length 2, etc.    │
  │  → The FIRST time it reaches destination = shortest path     │
  │  → No need to explore further!                               │
  │  → O(V + E) time!                                            │
  └──────────────────────────────────────────────────────────────┘

STEP 3: IDENTIFYING THE GRAPH PATTERN

The KEY Insight: BFS Level = Distance

  ┌────────────────────────────────────────────────────────────────┐
  │                                                                │
  │  Remember Problem 3 (BFS)?                                     │
  │    → BFS explores level-by-level                               │
  │    → Level 0: source (distance 0)                              │
  │    → Level 1: direct neighbors (distance 1)                    │
  │    → Level 2: neighbors of neighbors (distance 2)              │
  │    → Level k: all nodes at distance k                          │
  │                                                                │
  │  THE GUARANTEE:                                                │
  │    In an UNWEIGHTED graph, the FIRST time BFS reaches a node,  │
  │    that's via the SHORTEST path.                               │
  │                                                                │
  │  WHY?                                                          │
  │    → BFS processes all distance-1 nodes before distance-2      │
  │    → Processes all distance-2 before distance-3                │
  │    → So the FIRST time a node is discovered, it's at the      │
  │      MINIMUM possible distance from source                     │
  │    → No shorter path can exist (all shorter distances already  │
  │      fully explored)                                           │
  │                                                                │
  │  THIS IS THE MOST IMPORTANT PROPERTY OF BFS.                  │
  │                                                                │
  └────────────────────────────────────────────────────────────────┘

Proof by Contradiction (Intuitive)

  Suppose BFS first reaches node X at distance d.
  Could there be a shorter path of distance d' < d?
  
  → If such a path existed, it would go through nodes at 
    distances 1, 2, ..., d'-1
  → BFS already processed ALL nodes at distances 1 through d-1
  → So it would have discovered X at distance d' < d
  → But we said BFS first reached X at distance d
  → CONTRADICTION!
  
  Therefore: BFS distance = shortest distance. ∎
  
  THIS ONLY WORKS FOR UNWEIGHTED GRAPHS!
  
  Why it breaks with weights:
    Edge 0→1 costs 10, edge 0→2 costs 1, edge 2→1 costs 1
    BFS reaches 1 first via 0→1 (1 hop) with cost 10
    But 0→2→1 (2 hops) costs only 2 — SHORTER by cost!
    BFS doesn't know about costs — it only counts hops.
    
  For weighted graphs → Dijkstra (Problem 23)

What Changes from Problem 3's BFS?

  Problem 3 BFS:
    → Visit all reachable nodes
    → Record visit order
    
  Problem 9 BFS:
    → Visit all reachable nodes
    → Track DISTANCE from source for each node     ★ NEW ★
    → Stop when destination reached (early termination from Problem 8)
    → Return distance to destination

  The code is almost IDENTICAL to Problem 3.
  We just add a DISTANCE ARRAY.

Distance Array: The New Data Structure

  dist[node] = shortest distance from source to node
  
  Initialize:
    dist[source] = 0                    (distance to self = 0)
    dist[everything else] = -1 or ∞     (not yet reached)
  
  During BFS, when we discover a neighbor:
    dist[neighbor] = dist[current] + 1   (one hop further)
  
  This works because BFS guarantees:
    → current is at its SHORTEST distance
    → neighbor is discovered at current's distance + 1
    → this is neighbor's SHORTEST distance (first discovery)

Version 1: Shortest Distance Only

from collections import defaultdict, deque

def shortest_path_unweighted(n, edges, source, destination):
    """
    Find shortest path distance from source to destination
    in an UNWEIGHTED graph using BFS.
    
    WHY BFS works for shortest path (unweighted):
        → BFS explores nodes in order of DISTANCE from source
        → First time a node is reached = shortest distance
        → This is the level-order property from Problem 3
    
    WHY NOT DFS:
        → DFS explores one path deeply — may find a LONG path first
        → Must explore ALL paths to find shortest — exponential!
        → BFS finds shortest in O(V + E) — polynomial!
    
    Args:
        n: number of nodes
        edges: list of [u, v] pairs (undirected, unweighted)
        source: starting node
        destination: target node
    
    Returns:
        Shortest distance (number of edges), or -1 if unreachable
    """
    if source == destination:
        return 0
    
    # ─── BUILD GRAPH (Problem 1) ───
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)
    
    # ─── BFS WITH DISTANCE TRACKING ───
    visited = set()
    queue = deque([source])
    visited.add(source)
    dist = {source: 0}        # ★ distance array — THE key addition
    
    while queue:
        current = queue.popleft()
        
        for neighbor in graph[current]:
            if neighbor not in visited:
                visited.add(neighbor)
                dist[neighbor] = dist[current] + 1    # ★ one step further
                
                # Early termination (Problem 8 optimization)
                if neighbor == destination:
                    return dist[neighbor]
                
                queue.append(neighbor)
    
    # Destination never reached
    return -1

Version 2: Shortest Distance to ALL Nodes (Single Source All Destinations)

from collections import defaultdict, deque

def shortest_distances_all(n, edges, source):
    """
    Find shortest distance from source to EVERY other node.
    
    This is "Single Source Shortest Path" (SSSP) for unweighted graphs.
    
    Returns:
        dist: dictionary mapping each reachable node to its distance
              unreachable nodes have distance -1
    
    This is the BFS version of what Dijkstra (Problem 23) does 
    for weighted graphs.
    """
    # Build graph (Problem 1)
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)
    
    # BFS (Problem 3) with distance tracking
    dist = {i: -1 for i in range(n)}    # initialize all as unreachable
    dist[source] = 0
    
    visited = set([source])
    queue = deque([source])
    
    while queue:
        current = queue.popleft()
        
        for neighbor in graph[current]:
            if neighbor not in visited:
                visited.add(neighbor)
                dist[neighbor] = dist[current] + 1
                queue.append(neighbor)
    
    return dist

Version 3: Shortest Path WITH Path Reconstruction

from collections import defaultdict, deque

def shortest_path_with_route(n, edges, source, destination):
    """
    Find shortest path AND the actual route.
    
    Uses PARENT TRACKING (previewed in Problem 8):
        → For each discovered node, record WHO discovered it
        → Trace back from destination to source via parents
        → Reverse to get source-to-destination path
    
    Returns:
        (distance, path) or (-1, []) if unreachable
    """
    if source == destination:
        return (0, [source])
    
    # Build graph
    graph = defaultdict(list)
    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)
    
    # BFS with parent tracking
    visited = set([source])
    queue = deque([source])
    dist = {source: 0}
    parent = {source: None}       # ★ who discovered each node?
    
    while queue:
        current = queue.popleft()
        
        for neighbor in graph[current]:
            if neighbor not in visited:
                visited.add(neighbor)
                dist[neighbor] = dist[current] + 1
                parent[neighbor] = current        # ★ record parent
                
                if neighbor == destination:
                    # ─── RECONSTRUCT PATH ───
                    path = []
                    node = destination
                    while node is not None:
                        path.append(node)
                        node = parent[node]
                    path.reverse()
                    return (dist[destination], path)
                
                queue.append(neighbor)
    
    return (-1, [])

How Path Reconstruction Works

  During BFS from 0:
    parent[0] = None  (source)
    parent[1] = 0     (discovered from 0)
    parent[2] = 0     (discovered from 0)
    parent[3] = 1     (discovered from 1)
    parent[4] = 1     (discovered from 1)
    parent[5] = 2     (discovered from 2)
    parent[6] = 3     (discovered from 3)
    parent[7] = 5     (discovered from 5)

  Trace back from 7:
    7 → parent[7] = 5
    5 → parent[5] = 2
    2 → parent[2] = 0
    0 → parent[0] = None → STOP

  Backward chain: 7 ← 5 ← 2 ← 0
  Reverse:        0 → 2 → 5 → 7  ✓

  ┌──────────────────────────────────────────────────────────┐
  │  Parent tracking gives us the BFS TREE:                   │
  │                                                           │
  │              0 (root)                                     │
  │             / \                                           │
  │            1   2                                          │
  │           / \   \                                         │
  │          3   4   5                                        │
  │          |       |                                        │
  │          6       7                                        │
  │                                                           │
  │  Following parent pointers from any node back to root     │
  │  gives the SHORTEST PATH to that node.                    │
  └──────────────────────────────────────────────────────────┘

Version 4: Shortest Path on a GRID

from collections import deque

def shortest_path_grid(grid, sr, sc, dr, dc):
    """
    Shortest path on a grid from (sr,sc) to (dr,dc).
    
    Combines:
        → Problem 6: grid as graph
        → Problem 9: BFS for shortest path
    
    Grid: 1 = open, 0 = wall
    Returns: shortest distance, or -1 if unreachable
    """
    if grid[sr][sc] == 0 or grid[dr][dc] == 0:
        return -1
    
    if sr == dr and sc == dc:
        return 0
    
    rows, cols = len(grid), len(grid[0])
    directions = [(-1, 0), (1, 0), (0, -1), (0, 1)]
    
    visited = set()
    queue = deque([(sr, sc, 0)])    # (row, col, distance)
    visited.add((sr, sc))
    
    while queue:
        r, c, d = queue.popleft()
        
        for delta_r, delta_c in directions:
            nr, nc = r + delta_r, c + delta_c
            
            if (0 <= nr < rows and 0 <= nc < cols 
                and (nr, nc) not in visited 
                and grid[nr][nc] == 1):
                
                if nr == dr and nc == dc:
                    return d + 1        # found destination!
                
                visited.add((nr, nc))
                queue.append((nr, nc, d + 1))
    
    return -1

STEP 6: COMPLEXITY ANALYSIS

Time Complexity: O(V + E)

  IDENTICAL to BFS from Problem 3!

  WHY?
    → We added ONE extra operation per node: dist[neighbor] = dist[current] + 1
    → This is O(1) per neighbor — doesn't change overall complexity
    → Each node still enqueued/dequeued once: O(V)
    → Each edge still checked once: O(E)
    → Total: O(V + E)

  With early termination:
    → Best case: O(1) if source == destination
    → Average: O(V + E) / 2 if destination is "halfway"
    → Worst case: O(V + E) if destination unreachable
    → Still O(V + E) asymptotically

  For grid: V = R × C, E ≈ 2 × R × C → O(R × C)

Space Complexity: O(V)

  1. Adjacency list: O(V + E)    (input)
  2. Visited set: O(V)
  3. Queue: O(V)
  4. Distance dict: O(V)          ★ new — but still O(V)
  5. Parent dict: O(V)            (if path reconstruction needed)
  
  Extra space beyond input: O(V)

Comparison with Brute Force (DFS All Paths)

  ┌──────────────────────┬──────────────┬──────────────────────┐
  │  Approach            │  Time         │  Correct?            │
  ├──────────────────────┼──────────────┼──────────────────────┤
  │  BFS                 │  O(V + E)     │  YES (guaranteed)    │
  │  DFS (find any path) │  O(V + E)     │  NO (not shortest)   │
  │  DFS (all paths)     │  O(V!)        │  YES (but too slow)  │
  └──────────────────────┴──────────────┴──────────────────────┘

  BFS is the ONLY efficient algorithm for shortest path 
  in unweighted graphs.

STEP 7: WHY DFS FAILS — Concrete Proof

  Graph:
      0 --- 1 --- 2
      |           |
      |           |
      5 --- 4 --- 3

  Shortest path from 0 to 3: 0 → 1 → 2 → 3 (distance 3)

  DFS from 0 (neighbors in order [1, 5]):
    DFS(0) → DFS(5) → DFS(4) → DFS(3) → FOUND!
    Path: 0 → 5 → 4 → 3  (distance 3)
    
  Lucky! Same distance here. But consider:

      0 --- 1 --- 2 --- 3
      |                  |
      5 --- 6 --- 7 --- 8

  Shortest: 0 → 1 → 2 → 3 (distance 3)

  DFS from 0 (neighbors in order [1, 5]):
    DFS(0) → DFS(1) → DFS(2) → DFS(3) → FOUND!
    Path: 0 → 1 → 2 → 3 (distance 3) — lucky again!

  BUT if neighbors are ordered differently: [5, 1]:
    DFS(0) → DFS(5) → DFS(6) → DFS(7) → DFS(8) → DFS(3) → FOUND!
    Path: 0 → 5 → 6 → 7 → 8 → 3 (distance 5) — WRONG!
    
  DFS path depends on neighbor ORDER, not on DISTANCE.
  BFS path depends on DISTANCE, regardless of neighbor order.
  
  ┌──────────────────────────────────────────────────────────────┐
  │  BFS GUARANTEE:                                              │
  │    First discovery of a node = shortest distance             │
  │    Regardless of graph structure or neighbor ordering         │
  │                                                              │
  │  DFS NON-GUARANTEE:                                          │
  │    First discovery of a node = some distance                 │
  │    Could be shortest, could be longest, depends on luck      │
  └──────────────────────────────────────────────────────────────┘

STEP 8: WHAT IF THE GRAPH WERE...

What if Directed?

  BFS still works for directed graphs!
  
  Just build directed adjacency list:
    graph[u].append(v)    ← only u→v, not v→u
  
  BFS explores only reachable nodes following edge directions.
  Distance is still shortest in terms of hops.
  
  Example:
    0 → 1 → 2 → 3
    0 → 3              (direct edge)
  
  BFS from 0:
    Level 0: {0}
    Level 1: {1, 3}     ← 3 reached directly!
    
  dist[3] = 1 (via direct edge 0→3)
  Even though 0→1→2→3 exists (distance 3), BFS finds the shorter one.

What if Weighted?

  ★ BFS DOES NOT WORK FOR WEIGHTED SHORTEST PATH! ★
  
  Example:
      0 ---(1)--- 1 ---(1)--- 2
      |                        |
      +-------(10)-------------+

  Edge costs in parentheses.
  
  BFS (counting hops):
    Level 0: {0}
    Level 1: {1, 2}     ← 2 reached via direct edge (cost 10)
    dist[2] = 1 hop
    
  But actual shortest PATH BY COST:
    0 → 1 → 2 = cost 1 + 1 = 2
    0 → 2     = cost 10
    
  Shortest by cost is 0→1→2 (cost 2), but BFS said 0→2 (1 hop).
  BFS is right about HOPS but wrong about COST!

  For weighted shortest path → Dijkstra's algorithm (Problem 23)
  
  ┌──────────────────────────────────────────────────────────────┐
  │  BFS = shortest path when all edges have SAME cost           │
  │  Dijkstra = shortest path when edges have DIFFERENT costs    │
  │  Bellman-Ford = shortest path even with NEGATIVE costs       │
  │                                                              │
  │  This is the algorithm selection hierarchy for shortest path.│
  │  BFS is the simplest case (cost = 1 for all edges).         │
  └──────────────────────────────────────────────────────────────┘

What if We Need ALL Shortest Paths (Not Just One)?

  Multiple shortest paths might exist:
    0 → 1 → 3 → 6 → 7  (distance 4)
    0 → 1 → 4 → 6 → 7  (distance 4)
    
  Both have distance 4 — both are shortest from 0 to 7 via 6.
  
  To find ALL shortest paths:
    → Don't stop at first discovery
    → Allow multiple parents per node (if same distance)
    → parent becomes a LIST: parent[node] = [all nodes that discovered it at min dist]
    → Reconstruct all paths using backtracking on parent lists
    
  This is more advanced — for now, one shortest path suffices.

STEP 10: THE SHORTEST PATH ALGORITHM HIERARCHY

  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                    │
  │  WHICH SHORTEST PATH ALGORITHM DO I USE?                          │
  │                                                                    │
  │  ┌─────────────────────────────────────────────────┐              │
  │  │  Are all edge weights equal (or unweighted)?     │              │
  │  │                                                   │              │
  │  │  YES → BFS  ★ Problem 9 (this problem) ★        │              │
  │  │         O(V + E)                                  │              │
  │  │                                                   │              │
  │  │  NO → Are there negative weights?                 │              │
  │  │       │                                           │              │
  │  │       ├── NO → Dijkstra (Problem 23)              │              │
  │  │       │        O((V + E) log V)                   │              │
  │  │       │                                           │              │
  │  │       └── YES → Bellman-Ford (Problem 26)         │              │
  │  │                  O(V × E)                          │              │
  │  │                                                   │              │
  │  │  Need ALL pairs? → Floyd-Warshall (Problem 33)    │              │
  │  │                     O(V³)                          │              │
  │  └─────────────────────────────────────────────────┘              │
  │                                                                    │
  │  BFS is the SIMPLEST and FASTEST shortest path algorithm.         │
  │  It works for the SIMPLEST case: all edges equal weight.          │
  │  Each subsequent algorithm handles MORE general cases              │
  │  at the cost of more complexity.                                   │
  │                                                                    │
  └──────────────────────────────────────────────────────────────────┘

STEP 11: HOW I WOULD RECOGNIZE THIS PATTERN IN AN INTERVIEW

  ┌────────────────────────────────────────────────────────────────┐
  │                                                                │
  │  SIGNAL 1: "Minimum number of steps/moves/hops"                │
  │    → SHORTEST PATH                                             │
  │    → Each step = 1 unit → unweighted → BFS                    │
  │                                                                │
  │  SIGNAL 2: "Shortest path" + all edges same cost               │
  │    → BFS (not Dijkstra!)                                       │
  │    → Simpler and faster than Dijkstra for this case            │
  │                                                                │
  │  SIGNAL 3: "Minimum transformations/changes"                   │
  │    → Often an implicit graph problem                           │
  │    → Each transformation = one edge → BFS                      │
  │    → Example: Word Ladder (Problem 15)                         │
  │                                                                │
  │  SIGNAL 4: Grid + "shortest path" + no weights                 │
  │    → BFS on grid (Version 4)                                   │
  │    → Each cell transition = 1 step                             │
  │                                                                │
  │  THINKING PROCESS:                                             │
  │    "I see 'shortest/minimum' + equal costs"                    │
  │    → "BFS explores nearest first"                              │
  │    → "First discovery = shortest distance"                     │
  │    → "Use BFS with distance array"                             │
  │    → "If I also need the path: add parent tracking"            │
  │                                                                │
  │  RED FLAGS (don't use BFS):                                    │
  │    "Edges have different weights" → Dijkstra (Problem 23)      │
  │    "Negative edge weights" → Bellman-Ford (Problem 26)         │
  │    "All pairs shortest" → Floyd-Warshall (Problem 33)          │
  │                                                                │

STEP 12: KEY TAKEAWAY

  ╔════════════════════════════════════════════════════════════════════╗
  ║                                                                    ║
  ║   NEW CONCEPT LEARNED: BFS = SHORTEST PATH (UNWEIGHTED)          ║
  ║                                                                    ║
  ║   1. BFS explores nodes in ORDER OF DISTANCE from source          ║
  ║      → First discovery = shortest distance (GUARANTEED)           ║
  ║      → This is THE most important property of BFS                 ║

  ║   2. DISTANCE ARRAY: dist[node] = dist[parent] + 1               ║
  ║      → The ONLY addition to basic BFS (Problem 3)                 ║
  ║      → Tracks shortest distance to every reachable node           ║
  ║                                                                    ║
  ║   3. PATH RECONSTRUCTION: parent tracking + trace back            ║
  ║      → Record who discovered each node                            ║
  ║      → Follow parents from destination back to source             ║
  ║      → Reverse to get source → destination path                   ║
  ║                                                                    ║
  ║   4. DFS CANNOT find shortest path efficiently                    ║
  ║      → DFS explores deep, not nearest-first                       ║
  ║      → Would need to try ALL paths → exponential                  ║
  ║                                                                    ║
  ║   5. BFS only works for UNWEIGHTED (or equal-weight) graphs      ║

  ║      → This is the first step in the shortest path hierarchy      ║
  ║                                                                    ║

  ║    the ripple reaches distance-1 before distance-2.               ║

  ║                                                                    ║
