Problem 1: Build a Graph (Adjacency List)

[Algorithm Family: Graph Representation]

[Phase 1: Fundamentals — "I can represent and traverse any graph"]

PROBLEM STATEMENT

You are building a social network. Given N users (numbered 0 to N-1)
and a list of friendships (pairs of users who are friends),
store this information so that you can quickly answer:
  → "Who are the friends of user X?"

Input:
  n = 5  (users: 0, 1, 2, 3, 4)
  edges = [[0,1], [0,2], [1,3], [2,3], [3,4]]

Output:
  A data structure where you can look up any user's friends instantly.
  
  Friends of 0 → [1, 2]
  Friends of 1 → [0, 3]
  Friends of 2 → [0, 3]
  Friends of 3 → [1, 2, 4]
  Friends of 4 → [3]

    |      |
    2 ---- 3
           |
           |
           4

  Edge List (what we're given):
    0 — 1
    0 — 2
    1 — 3
    2 — 3
    3 — 4

╚══════════════════╧══════════════╧═════════════════════════╝

  Dense means: edges close to V*(V-1)/2 (undirected) or V*(V-1) (directed)
  Sparse means: edges close to V or much less than V²
  Here: 5 edges vs max 10 → SPARSE

What Does "Undirected" Mean Visually?

  UNDIRECTED: 0 ---- 1     (friendship goes BOTH ways)
              0 is friend of 1  AND  1 is friend of 0

  DIRECTED:   0 ---→ 1     (one-way relationship)
              0 follows 1  BUT  1 may NOT follow 0
              (like Twitter followers vs Facebook friends)

STEP 2: BRUTE FORCE FIRST — What's the Naive Approach?

Approach 1: Store the raw edge list

  edges = [[0,1], [0,2], [1,3], [2,3], [3,4]]

  Query: "Who are friends of node 3?"
  → Scan EVERY edge
  → Check if 3 appears in [0,1]? NO
  → Check if 3 appears in [0,2]? NO
  → Check if 3 appears in [1,3]? YES → friend is 1
  → Check if 3 appears in [2,3]? YES → friend is 2
  → Check if 3 appears in [3,4]? YES → friend is 4
  → Answer: [1, 2, 4]

Why Brute Force Fails

  Every query scans ALL edges → O(E) per query
  
  If you have 1 million users and 10 million friendships:
    → Each "who are my friends?" costs scanning 10 million edges
    → If 1 million users ask → 10^6 × 10^7 = 10^13 operations
    → IMPOSSIBLY SLOW

  We need: O(1) to ACCESS a user's friend list
           O(degree) to ITERATE through friends
           (degree = number of friends that specific user has)

STEP 3: IDENTIFYING THE GRAPH PATTERN

What Representation Fits?

  We need to answer: "Give me all neighbors of node X" — FAST
  
  Three options exist:

  ┌─────────────────┬───────────────────┬──────────────────┐
  │  Representation  │  "Friends of X?"  │  Space           │
  ├─────────────────┼───────────────────┼──────────────────┤
  │  Edge List       │  O(E) — scan all  │  O(E)            │
  │  Adjacency Matrix│  O(V) — scan row  │  O(V²)           │
  │  Adjacency List  │  O(degree) — BEST │  O(V + E)        │
  └─────────────────┴───────────────────┴──────────────────┘

  For SPARSE graphs (E << V²):
    → Adjacency List WINS on both time and space
    → This social network: O(5+5) = O(10) vs O(25) for matrix
  
  For DENSE graphs (E ≈ V²):
    → Matrix might win (discussed in Problem 2)

What is an Adjacency List?

  A dictionary (or array of lists) where:
    KEY   = a node
    VALUE = list of all nodes connected to it

  Think of it as:
    "Each person carries a CONTACT LIST of their friends"

  For our graph:
    {
      0: [1, 2],        ← node 0 connects to 1 and 2
      1: [0, 3],        ← node 1 connects to 0 and 3
      2: [0, 3],        ← node 2 connects to 0 and 3
      3: [1, 2, 4],     ← node 3 connects to 1, 2, and 4
      4: [3]            ← node 4 connects to 3 only
    }

Why Adjacency List Works Here

  → SPARSE graph: only 5 edges, 5 nodes
  → Space: O(V + 2E) for undirected = O(5 + 10) = O(15)
    (each edge stored TWICE — once for each endpoint)
  → Query "friends of 3": just return graph[3] → O(1) access + O(degree) iteration
  → Adding a new friendship: append to both lists → O(1)

STEP 4: ALGORITHM WALKTHROUGH — Building the Adjacency List

The Core Idea

  For each edge [u, v] in the edge list:
    → Add v to u's neighbor list    (u can reach v)
    → Add u to v's neighbor list    (v can reach u)  ← ONLY for undirected!

STEP 5: COMPLETE CODE

Version 1: Undirected Graph — Using defaultdict

from collections import defaultdict

def build_undirected_graph(n, edges):
    """
    Build an adjacency list for an UNDIRECTED graph.
    
    Args:
        n: number of nodes (0 to n-1)
        edges: list of [u, v] pairs representing edges
    
    Returns:
        graph: dictionary mapping each node to its list of neighbors
    
    Key insight: each edge [u, v] means BOTH u→v AND v→u
    """
    # defaultdict(list) → accessing a missing key auto-creates empty list
    # This avoids KeyError when a node has no edges yet
    graph = defaultdict(list)
    
    # Ensure all nodes exist, even isolated ones (no edges)
    for node in range(n):
        graph[node]  # just accessing creates the empty list entry
    
    # Process each edge
    for u, v in edges:
        graph[u].append(v)  # u can reach v
        graph[v].append(u)  # v can reach u (UNDIRECTED = bidirectional)
    
    return graph


# ─── USAGE ───
n = 5
edges = [[0, 1], [0, 2], [1, 3], [2, 3], [3, 4]]

graph = build_undirected_graph(n, edges)

# Print the adjacency list
print("Adjacency List:")
for node in range(n):
    print(f"  Node {node} → {list(graph[node])}")

# Query: friends of node 3
print(f"\nFriends of node 3: {graph[3]}")

# Query: degree of each node (number of connections)
print("\nDegree of each node:")
for node in range(n):
    print(f"  Node {node}: degree = {len(graph[node])}")

Version 2: Directed Graph — "Twitter follows"

from collections import defaultdict

def build_directed_graph(n, edges):
    """
    Build an adjacency list for a DIRECTED graph.
    
    Args:
        n: number of nodes (0 to n-1)
        edges: list of [u, v] pairs where u → v (u follows v)
    
    Returns:
        graph: dictionary mapping each node to nodes it points TO
    
    Key difference from undirected:
        Edge [u, v] means ONLY u→v, NOT v→u
    """
    graph = defaultdict(list)
    
    for node in range(n):
        graph[node]
    
    for u, v in edges:
        graph[u].append(v)  # u points to v
        # graph[v].append(u)  ← THIS LINE IS REMOVED for directed!
    
    return graph


# ─── USAGE ───
n = 5
# "0 follows 1", "0 follows 2", "1 follows 3", "2 follows 3", "3 follows 4"
edges = [[0, 1], [0, 2], [1, 3], [2, 3], [3, 4]]

graph = build_directed_graph(n, edges)

print("Directed Adjacency List:")
for node in range(n):
    print(f"  Node {node} → {list(graph[node])}")

Version 3: Weighted Graph — "Flight costs"

from collections import defaultdict

def build_weighted_graph(n, edges):
    """
    Build an adjacency list for a WEIGHTED DIRECTED graph.
    
    Args:
        n: number of nodes
        edges: list of [u, v, weight] — edge from u to v with given weight
    
    Returns:
        graph: dict mapping node → list of (neighbor, weight) tuples
    
    Real-world: cities connected by flights with different costs
    """
    graph = defaultdict(list)
    
    for node in range(n):
        graph[node]
    
    for u, v, weight in edges:
        graph[u].append((v, weight))  # store BOTH neighbor AND weight
        # For undirected weighted: also add graph[v].append((u, weight))
    
    return graph


# ─── USAGE ───
n = 4
# [from, to, cost]
edges = [[0, 1, 100], [0, 2, 300], [1, 2, 50], [2, 3, 200]]

graph = build_weighted_graph(n, edges)

print("Weighted Directed Adjacency List:")
for node in range(n):
    print(f"  Node {node} → {list(graph[node])}")
    for neighbor, weight in graph[node]:
        print(f"         to {neighbor}, cost = {weight}")

Output:

Weighted Directed Adjacency List:
  Node 0 → [(1, 100), (2, 300)]
         to 1, cost = 100
         to 2, cost = 300
  Node 1 → [(2, 50)]
         to 2, cost = 50
  Node 2 → [(3, 200)]
         to 3, cost = 200
  Node 3 → []

Version 4: Using Plain Dictionary (No imports)

def build_graph_plain(n, edges, directed=False):
    """
    Build adjacency list without any imports.
    
    Sometimes in interviews, you want minimal setup.
    """
    # Initialize empty list for each node
    graph = {i: [] for i in range(n)}
    
    for u, v in edges:
        graph[u].append(v)
        if not directed:
            graph[v].append(u)
    
    return graph


# ─── USAGE ───
graph = build_graph_plain(5, [[0,1],[0,2],[1,3],[2,3],[3,4]])
print(graph)
# {0: [1, 2], 1: [0, 3], 2: [0, 3], 3: [1, 2, 4], 4: [3]}

STEP 6: COMPLEXITY ANALYSIS

Time Complexity: O(V + E)

  ┌──────────────────────────────────────────────────────────┐
  │  Operation                    │  Time                    │
  ├──────────────────────────────┼──────────────────────────┤
  │  Initialize V empty lists     │  O(V)                    │
  │  Process E edges              │  O(E) — one append each  │
  │  (undirected: 2 appends/edge) │  Still O(E)              │
  │                               │                          │
  │  TOTAL BUILD TIME             │  O(V + E)                │
  ├──────────────────────────────┼──────────────────────────┤
  │  Query "neighbors of X"       │  O(1) dict lookup        │
  │  Iterate neighbors of X       │  O(degree(X))            │
  │  Check "is Y neighbor of X?"  │  O(degree(X)) — scan list│
  └──────────────────────────────┴──────────────────────────┘

Space Complexity: O(V + E)

  WHY O(V + E)?
  
  → V entries in the dictionary (one per node)
  → For undirected: each edge stored twice → 2E neighbor entries
  → For directed: each edge stored once → E neighbor entries
  → Total: O(V + E)  [we drop constants like 2]

  Concrete example:
    V = 5, E = 5 (undirected)
    → 5 dictionary keys + 10 list entries = 15 total storage units
    
  Compare with adjacency matrix:
    → O(V²) = O(25) — stores 25 cells even though only 5 edges exist
    → WASTED SPACE for sparse graphs!

Why O(V + E) and Not O(V × E)?

  CRITICAL INSIGHT:
  
  We process each edge ONCE (or twice for undirected)
  We do NOT process each edge for each vertex
  
  It's a SINGLE PASS through the edge list:
    for u, v in edges:     ← this loop runs E times, NOT V×E times
        graph[u].append(v)  ← O(1) per append
        graph[v].append(u)  ← O(1) per append
  
  Total: E iterations × O(1) work = O(E)
  Plus initialization: O(V)
  Grand total: O(V + E)

STEP 7: WHAT IF THE GRAPH WERE...

What if Directed vs Undirected?

  UNDIRECTED (friendship):
    → Each edge [u,v] → add BOTH directions
    → graph[u].append(v) AND graph[v].append(u)
    → Total entries in all lists = 2 × E
    
  DIRECTED (Twitter follow):
    → Each edge [u,v] → add ONE direction only
    → graph[u].append(v) ONLY
    → Total entries in all lists = E
    
  CODE DIFFERENCE: literally ONE line

What if Weighted vs Unweighted?

  UNWEIGHTED:
    → graph[u].append(v)
    → Store just the neighbor
    
  WEIGHTED:
    → graph[u].append((v, weight))
    → Store neighbor AND weight as tuple
    
  This changes how we ITERATE:
    Unweighted: for neighbor in graph[node]:
    Weighted:   for neighbor, weight in graph[node]:

What if Nodes Aren't Numbers?

  Nodes can be ANYTHING hashable: strings, tuples, etc.
  
  Example: Cities as nodes
    graph = defaultdict(list)
    graph["NYC"].append("LA")
    graph["NYC"].append("Chicago")
    
  Example: Grid cells as nodes (Problem 6 preview)
    graph[(0,0)].append((0,1))
    graph[(0,0)].append((1,0))
